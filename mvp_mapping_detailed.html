How this guide should be used

Use it as a truthier inventory, not as a sign-off doc. The key question is not “does some ticket exist?” but “is this need clearly owned by product/dev work, and is that work aligned to the right cohort?”

Important rule used here: placeholder initiatives do not count as coverage.

Method

  • Start with the official Phase 1 MVP Requirements Map for direct requirement-to-ticket links.
  • Use the MVP Needs for MoT doc comments to capture discovery answers and open questions.
  • Credit non-Jira artifacts when they genuinely answer a need.
  • Refresh Jira statuses/fix versions to catch drift from the older crosswalk.

Counts

  • 31 unique explicit ticket references used in the matrix
  • 30 unique adjacent / inferred ticket references used as supporting context
  • 6 needs materially pushed to later cohorts or placeholder buckets

Key concerns before anyone treats this as coverage proof

  1. BOOK-372 is the biggest drift item: the requirement map points to bank reconciliation, but live Jira now has it in POP:R03 New Cohort, not the initial cohort.
  2. The official map contains semantically weak reporting links: 13a.i maps to WIRE-1357 (Voids) and 13a.ii maps to WIRE-1356 (Reversals).
  3. Architecture evidence exists (diagram + slide deck), so P0.1 is not an “evidence gap”; it is an ownership / delivery-ticket gap.
  4. Placeholder initiatives (BOOK-736/737/738 and WIRE-1703/1705/1706/1708/1709) should not be counted as real requirement coverage.
  5. Several future-facing tax, garnishment, and onboarding needs still have no explicit owned tickets in the current map.
